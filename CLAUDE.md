# CLAUDE.md

Guidance for Claude Code working in this repo. Read this before changing anything.

## What this is

**Pinch** â€” drive a real Claude Code session from an Apple Watch over cellular.
It is a **tether to this Mac**, not a cloud product: the backend runs the Claude
Agent SDK against the live repos under `PINCH_PROJECT_ROOTS` (e.g.
`~/Desktop/projects`) and a tunnel exposes it to the watch. The watch is a thin
client. The cloud mode in `infra/cloud` is a different thing â€” it only sees
git-pushed code and is not remote control of this Mac.

The repo is **public** (`github.com/JoshKappler/apple-watch-claude-code-`).

### Two apps, one shared core (the part to NOT break)

The `watch/` Xcode project (`watch/project.yml`) has **two app targets**: `Pinch` (watchOS)
and `PinchPhone` (iOS). They are **independent** clients of the same backend â€” NOT a paired
WatchKit companion. The backend is the only sync point. `watch/Sources/` is split:

- **`Sources/Shared/`** â€” compiled by BOTH targets. The whole brain lives here: `Protocol`,
  `WSClient` (the HTTP transport / durable outbox / resume / poll-cursor engine â€” named
  `WSClient` for history, it's HTTP), `Store` (`PinchStore`), `Speaker`, `ShakeDetector`,
  `DictationIntent`, `PushRegistration`, `DeviceID`, `Haptics` (cross-platform via `#if os`),
  and `Secrets.swift` (gitignored). **Editing anything here affects BOTH apps** â€” keep it
  cross-platform (guard platform APIs with `#if os(watchOS)` / `#else`) and build BOTH targets
  before trusting a change. The one platform fork inside Shared is in `WSClient.openSession`:
  the phone adds `render:"rich"` to the `/api/session` body via `#if !os(watchOS)`; the watch
  omits it â†’ backend defaults to `plain` â†’ the watch request is byte-for-byte unchanged.
- **`Sources/Watch/`** â€” watchOS-only: `PinchApp` (@main), `CrownControls`, `Dictation`
  (WatchKit `presentTextInputController`), the watch `Views/`, `Assets.xcassets`.
- **`Sources/Phone/`** â€” iOS-only: `PinchPhoneApp` (@main), the phone view layer (Fleet list â†’
  Conversation â†’ Composer, Settings, Project picker, Permission sheet), `PinchTheme`, the
  from-scratch `Markdown`/`DiffView` renderers, `PhoneDictation` (SFSpeechRecognizer â€” works on
  iOS, unlike watchOS), `Assets.xcassets`.

Render mode (`packages/protocol` `RenderMode` = `plain` | `rich`) is per-session, set at
creation by the client: phone sessions get `PHONE_SYSTEM_APPEND` (markdown/code/diffs OK),
watch sessions get `WATCH_SYSTEM_APPEND` (plain text). Default is `plain`, persisted in
`sessionStore`, so existing/watch sessions never change. The two appends share the SAME final
git-rule paragraph verbatim (`backend/src/session.ts`) â€” keep them in sync if you edit it.

Per-agent live status in the phone Fleet list is currently the FOCUSED agent only (the Store
parks non-focused agents, same as the watch switcher). Full live badges across the fleet want
a small additive backend change: include each session's `AgentState` in `GET /api/agents`.

npm workspaces at the root: `packages/protocol`, `backend`, `simulator`. The
watch is a separate native build.

## Transport â€” the thing that's easy to get wrong

There are **two transports** to the same session lifecycle:

- **The physical watch uses HTTP**, not WebSockets. watchOS refuses
  `URLSessionWebSocketTask` on the watch's network path, so the watch does
  request/response + a ~1.2s short-poll loop against `/api/*`
  (`backend/src/httpApi.ts`). The Swift client class is still named `WSClient`
  (`watch/Sources/Shared/WSClient.swift`) for historical reasons â€” it is HTTP.
- **The browser simulator uses the WebSocket** at `/ws` (`backend/src/wsServer.ts`,
  `connection.ts`). `PROTOCOL.md` describes this path.

Both share the session map, agent wiring, resume rule, event log, and idle sweep
via `backend/src/sessionRegistry.ts` â€” nothing is duplicated. The watch enters a
`wss://` URL; the app rewrites it to `https://` internally. Every watch request
carries `Authorization: Bearer <PINCH_TOKEN>` and `ngrok-skip-browser-warning`.

### Poll-cursor invariant (do NOT regress â€” this was the duplicate-bubble bug)

On a **resumed** session the watch must CONTINUE its poll cursor, never reset to
0 â€” the backend event log still holds the whole history, so a from-zero cursor
re-delivers everything and every assistant bubble appears twice. Cursor is
persisted to UserDefaults (`pinch.resumeCursor.<deviceId>`).

Exception: a **revived** session (backend restarted / idle-swept, rebuilt from
the durable record) restarts its event log at index 0, so the backend signals
`resetCursor:true` in the `/api/session` response and the watch zeroes its
cursor. The rule is: **continue cursor only when `resumed && !resetCursor`;
otherwise reset to 0.** Keep this branch in `openSession()`.

### Durable delivery + resume

- Prompts go into a **durable persisted outbox** in `WSClient.swift`, removed
  only on a confirmed 2xx, drained FIFO single-flight. Bubbles show
  Sendingâ€¦/Sent/Not sent. Backend dedups by client `promptId` in
  `httpApi.handlePrompt` so an at-least-once retry can't double-run a turn.
- Live `SessionState` is in-memory, so a restart/idle-sweep would wipe context.
  `backend/src/sessionStore.ts` durably records `{our id â†’ SDK session id +
  project/model/effort/mode/device}` to `backend/.pinch-sessions.json`. On
  resume, `reviveSession()` rebuilds the session with `options.resume = sdkId` so
  the SDK reloads the transcript and Claude keeps context.

## Watch-aware agent output

Every real session appends `WATCH_SYSTEM_APPEND` (`backend/src/session.ts`) to the
Claude Code preset system prompt (`systemPrompt: { type: "preset", preset:
"claude_code", append: â€¦ }`). It tells the agent it's talking to someone on an
Apple Watch â€” replies are read aloud (TTS) and shown on a tiny screen â€” so it must
write plain text (no Markdown/emoji/tables/code fences â€” they don't render and get
read aloud literally), keep replies brief, and offer choices as short numbered
prose. It shapes COMMUNICATION, plus one standing work rule â€” the origin-main git
default (no worktrees / no branch switching); tools/edits/rigor are otherwise
unchanged. It's a static string (prompt-cached with the preset). The mock session doesn't use it. If you
change how the watch renders/speaks replies, keep this append in sync.

## Build & verify

- **Whole TS workspace:** `npm run typecheck` (and `npm run build`) at the root.
- **Protocol-dist gotcha:** the backend imports `@pinch/protocol` from its built
  `dist/` (gitignored). After editing `packages/protocol/src/index.ts`, run
  `npm run build --workspace @pinch/protocol` BEFORE the backend typecheck/build
  will see new symbols (e.g. a new `srv.*` frame).
- **Watch app:**
  `cd watch && xcodebuild -project Pinch.xcodeproj -scheme Pinch -destination 'platform=watchOS Simulator,name=Apple Watch Ultra 3 (49mm)' build`.
  `Pinch.xcodeproj` is **gitignored** â€” generate it first with `xcodegen generate`
  (source of truth is `watch/project.yml`). Bundle id `com.joshua.kappler.pinch.watch`.
  **After editing any `Sources/Shared/` file, build the watch too** â€” a Shared change that
  breaks the watch is the main regression risk of the two-target split.
- **Phone app (`PinchPhone`, iOS, bundle id `com.joshua.kappler.pinch.phone`):** this Mac has
  the iOS SDKs but NOT the iOS *simulator runtime*, so a full packaged build can't finish here
  (actool can't device-thin the asset catalog without a runtime; install it with
  `xcodebuild -downloadPlatform iOS` or Xcode â†’ Settings â†’ Components to run on device/sim).
  To verify the CODE compiles + links without the runtime:
  `xcodebuild -project Pinch.xcodeproj -target PinchPhone -sdk iphonesimulator26.5 -configuration Debug CODE_SIGNING_ALLOWED=NO SYMROOT=/tmp/pp-sym OBJROOT=/tmp/pp-obj build`
  (use `-target`, not `-scheme` â€” `-scheme` fails to resolve a destination with no runtime).
  If actool blocks it, temporarily move `Sources/Phone/Assets.xcassets` aside + regenerate.
- **Incremental-build gotcha:** a plain `xcodebuild â€¦ build` sometimes reports
  BUILD SUCCEEDED without recompiling edited sources (stale incremental DB); the
  installed app then shows old behavior. The real code lives in
  `Pinch.app/Pinch.debug.dylib` (the `Pinch` binary is just a launcher stub whose
  mtime never changes). If the dylib mtime predates your edits, `clean build`,
  then `xcrun simctl uninstall` + `install`.
- **Flashing to the physical device â€” use `infra/flash-watch.sh`, never ad-hoc.**
  Every build is version `1.0 (1)`, so a flash is otherwise invisible: you can't tell
  a fresh binary from a stale one on the wrist. That, plus the wrong default Xcode
  (`xcode-select` is the stable Xcode, but an OS-27 watch/phone can ONLY be installed
  to with the **Xcode 27 beta** DDI) and several stray DerivedData trees you might
  install from, is why "fixes" kept not landing for a half-dozen reviews. The fix is a
  **build stamp compiled into the binary**: a gitignored `watch/Sources/Shared/BuildStamp.swift`
  (git SHA + `+`-if-dirty + timestamp) regenerated every build by the `Generate BuildStamp`
  preBuildScript in `project.yml` (needs `ENABLE_USER_SCRIPT_SANDBOXING: NO`), shown in
  **Settings â†’ Build** on both apps. `infra/flash-watch.sh [watch|phone] [--clean]` forces
  the beta toolchain, builds + installs from ONE path (`/tmp/pinch-flash-dd`), and prints the
  stamp it shipped. **Flash, then confirm Settings â†’ Build matches that stamp** â€” if it
  doesn't, the new code did NOT land (re-run with `--clean`). A stale binary cannot fake a
  fresh stamp. Don't reintroduce other `-derivedDataPath` install recipes (the old
  `/tmp/pinch-dd` one was a stale-install trap, now removed).
- **SourceKit false positives:** per-file diagnostics like "Cannot find type
  'ServerMsg'/'PinchStore'" are single-file-analysis noise. Trust xcodebuild.
- **Smoke test (WS path, mock):** `PINCH_MOCK=1 PINCH_TOKEN=test-token npm run dev`
  then `PINCH_TOKEN=test-token npm run smoke`.

## Operational gotchas (not obvious from code)

- **Restart the backend to apply backend changes.** The tether is a long-running
  `node dist/index.js`. Rebuilding `dist/` is NOT enough â€” the running process
  serves stale code until killed + relaunched (via `launchctl kickstart -k
  gui/$(id -u)/com.pinch.server`, or `npm run down && npm run up`).
  Tell: a brand-new session whose context ring reads ~25% full = a stale process.
  The watch can trigger this itself: **Settings â†’ Restart backend** POSTs
  `/api/restart` (`httpApi.handleRestart`), which spawns the detached
  `infra/restart-backend.sh`. That script builds FIRST (old process keeps serving),
  and only kills + relaunches if the build SUCCEEDS â€” a failed build leaves the
  tether alive. After the swap the watch 410s and `reviveSession()` restores the
  same conversation. Caveat: the button is itself backend code, so the very FIRST
  time you must restart from the Mac (the running process predates `/api/restart`).
- **Reinstall the watch app to apply watch changes.** All the client-side Swift
  only reaches the wrist on a fresh Xcode build + install.
- **Token sync.** After rotating `PINCH_TOKEN` in `backend/.env`, restart the
  backend â€” a long-running process holds the stale token and rejects the watch
  with `unauthorized`. The watch's `Secrets.token` must match the running value.

## Tunnel & launch

The watch URL is a **stable ngrok free static domain** (set as
`PINCH_NGROK_DOMAIN`). It never changes, so restarts don't strand the baked watch
build. ngrok free = one agent session at a time. Cloudflare named-tunnel scaffolding
exists in `infra/cloudflared/` but is unused (no domain).

- **`npm run up`** (`infra/launchd/install-launchd.sh`) â€” installs three
  LaunchAgents (`com.pinch.server`, `com.pinch.tunnel`, `com.pinch.watchdog`) into
  `~/Library/LaunchAgents` and loads them. They start at login (`RunAtLoad`),
  restart on crash (`KeepAlive` on non-clean exit), and the watchdog re-kicks or
  re-bootstraps anything wedged/booted-out every 120s â€” always-on. The installer
  reads `PINCH_NGROK_DOMAIN` from `backend/.env` to keep the URL fixed. Build first
  (`npm run build`); the installer does not build.
- **`npm run down`** (`infra/launchd/uninstall-launchd.sh`) â€” boots out + removes
  the agents.

## Secrets

Public repo + RCE-grade token, so secrets NEVER land in committed source.

- Backend: `backend/.env` (gitignored, keep `chmod 600`).
- Apps: `watch/Sources/Shared/Secrets.swift` (gitignored, shared by BOTH the watch and
  phone targets), referenced as `Secrets.serverURL` / `Secrets.token` from `PinchApp.swift`
  (watch) and `PinchPhoneApp.swift` (phone). Committed template is `watch/Secrets.example.swift`
  (outside `Sources/` so XcodeGen won't compile it). `setup.sh` creates both files and injects
  the token. **Never reintroduce a hardcoded token/URL into an app entry point** â€” and there is
  no launch-time force-write of serverURL/token, so a URL typed in Settings persists.
- Rotation is the real fix for a leak, not history rewriting (force-push does not
  fully erase a blob from GitHub). See `infra/SECURITY.md`.

## watchOS platform constraints (realities, not bugs)

- **Double-tap** (`.handGestureShortcut(.primaryAction)`) is Series 9 / Ultra 2+
  only. On-screen Send is the universal fallback.
- **No crown press** for apps â€” confirm/select is built from crown *rotation*
  (`CrownConfirm` threshold, `CrownPicker` dwell) in `CrownControls.swift`.
- **Voice = Apple system dictation** (`Dictation.swift`), not an always-on
  listener; `SFSpeechRecognizer` doesn't function on watchOS.
- **TTS** (`Speaker.swift`) is often silent without a Bluetooth route, so every
  spoken reply is paired with a haptic.
- **No background socket / connection.** Foreground only; reconnect + resume on
  reopen. APNs (`PushRegistration.swift`) is the re-engagement path and is a STUB.
- **Swipe gestures over a watchOS `ScrollView` never fire** â€” put swipe/collapse
  gestures on non-scroll surfaces. The caret editor is a sheet (not a nav push)
  so swipe-delete doesn't fight the system back-swipe.

## Conventions

User preferences live in the global `~/.claude/CLAUDE.md`. In short: build from
scratch over frameworks (no LangChain/n8n), get to working code fast, Zod at LLM
boundaries, tool_use for structured output, prompt caching on stable system
prompts. Git: one branch only â€” commit and push to `origin/main`; never create
worktrees, `checkout`/`switch` or create branches, or change the primary branch (the
watch can spawn several agents into one tree â€” they must not move it out from under
each other). Commit and push without asking once typecheck/tests pass; ask before
anything destructive. Skip trailing "what I changed" summaries â€” diffs are read
directly.
